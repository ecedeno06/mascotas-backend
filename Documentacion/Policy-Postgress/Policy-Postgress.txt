Eso es una política de seguridad de PostgreSQL llamada:

Row Level Security (RLS)

Es una de las funcionalidades MÁS poderosas de PostgreSQL para sistemas multitenant.

¿Qué hace?

Esta línea:

CREATE POLICY tenant_isolation
ON pets
USING (tenant_id = current_setting('app.tenant_id')::uuid);

le dice a PostgreSQL:

“Solo permite ver filas cuyo tenant_id
sea igual al tenant actual”
En otras palabras

Aunque hagas:

SELECT * FROM pets;

PostgreSQL automáticamente filtrará:

WHERE tenant_id = tenant_actual

SIN que tú lo escribas.

Primero debes activar RLS
ALTER TABLE pets
ENABLE ROW LEVEL SECURITY;
Luego creas la política
CREATE POLICY tenant_isolation
ON pets
USING (
   tenant_id = current_setting('app.tenant_id')::uuid
);
¿Qué es esto?
current_setting('app.tenant_id')

Es una variable de sesión PostgreSQL.

El backend debe establecerla

Ejemplo Node.js:

await db.query(
   `SET app.tenant_id = '${tenantId}'`
);
Flujo completo

Muy importante

RLS protege:

SELECT
También puedes proteger:
INSERT
WITH CHECK (...)
UPDATE
DELETE
Ejemplo completo
CREATE POLICY tenant_policy
ON pets
FOR ALL
USING (
   tenant_id = current_setting('app.tenant_id')::uuid
)
WITH CHECK (
   tenant_id = current_setting('app.tenant_id')::uuid
);
Entonces:
SELECT

Solo ve sus filas.

INSERT

Solo puede insertar con su tenant_id.

UPDATE

No puede modificar otra compañía.

DELETE

No puede borrar otra compañía.

Sin RLS

Tú dependes de esto:

WHERE tenant_id = ...

en TODOS los queries.

Con RLS

PostgreSQL se vuelve el guardia de seguridad.

Ejemplo real de problema SIN RLS

Developer escribe:

SELECT * FROM pets;

Oops.

Filtró todas las compañías.

Con RLS

PostgreSQL bloquea automáticamente.

Y ya tendrías aislamiento SaaS real a nivel PostgreSQL.
por que app. ?

Porque app. es simplemente un namespace personalizado de configuración en PostgreSQL.

Esto:
current_setting('app.id_compania')

significa:

“obtén la variable de sesión llamada app.id_compania”
PostgreSQL permite variables custom

Usando:

SET nombre_variable = valor;
Ejemplo
SET app.id_compania = '5';

Luego:

SELECT current_setting('app.id_compania');

devuelve:

5
¿Por qué usar app.?

Porque PostgreSQL recomienda que las variables custom tengan prefijo.

Ejemplo:

app.
myapp.
tenant.
security.
Evita conflictos internos

Porque PostgreSQL ya tiene configuraciones oficiales como:

work_mem
timezone
search_path

Entonces:

SET id_compania = 5

podría causar conflictos.

Por eso se usa namespace
SET app.id_compania = 5
app no tiene significado especial

Podría ser:

SET empresa.id_compania = 5

o:

SET tenant.current = 5
Pero normalmente se usa:
app.

como convención.

Flujo completo
Backend Node.js
await db.query(`
   SET app.id_compania = '${req.user.id_compania}'
`);
PostgreSQL guarda esa variable en la sesión actual
Luego RLS la usa
USING (
   id_compania = current_setting('app.id_compania')::BIGINT
)
Entonces PostgreSQL pregunta:
“¿La fila pertenece a la compañía actual?”
Si sí

✅ permite acceso

Si no

❌ bloquea la fila

Importante

La variable existe SOLO durante la conexión actual.

Por eso en pool connections debes setearla siempre

Cada request.

Ejemplo típico
Middleware
export const tenantMiddleware = async (req, res, next) => {

   await db.query(`
      SET app.id_compania = '${req.user.id_compania}'
   `);

   next();

};
Luego cualquier query queda protegido automáticamente
